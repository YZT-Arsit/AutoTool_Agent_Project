# AutoToolBench

An adaptive tool-calling agent benchmark under budget constraints.  
The goal is to improve upon a ReAct baseline by planning, replanning and reflecting, while minimizing tool calls and token usage.  
All code runs locally with a pluggable LLM interface (default `MockLLM`).

## Motivation

This project focuses on **non-RAG main course** agent research.  
Instead of retrieving documents, agents must plan and execute with local tools (SQLite, file I/O, python sandbox, log search, tests).  
By designing an `AdaptiveAgent` with planning, reflection and budget control we can achieve higher success rates and fewer costly tool invocations.

## Quick Start

```bash
cd autotoolbench
python -m autotoolbench make-data      # generate sample.db, logs, tasks
python -m autotoolbench eval --agent all --seed 0 --noise 0.0
```

See `data/tasks.jsonl` for the benchmark tasks.  After running, check `reports/latest_report.md` for results.

### Commands

```bash
# generate benchmark data
autotoolbench make-data

# run a single task
autotoolbench run --task-id T001 --agent adaptive --seed 0 --noise 0.2
# or positional
autotoolbench run T001 --agent adaptive

# evaluate agents on all tasks
autotoolbench eval --agent all --seed 0 --noise 0.0

# run ablation studies
autotoolbench ablate --seed 0 --noise 0.2
```

> Note: `run` command accepts `--task-id` option or positional argument.

## Metrics

* **Success rate**: fraction of tasks where validator passes
* **Avg tool calls** / **avg steps**: efficiency measures
* **Failure types**: breakdown for AdaptiveAgent (from `reflection` labels)
* **Trajectories**: detailed step-by-step trace saved under `data/logs/trajectory_{task}_{seed}.json`.

Ablation studies compare variants:
- **no_reflector**: disables reflection logic
- **no_budget**: removes budget constraints (effectively unlimited)
- **no_replan**: planning step occurs once only

Reports generated by `autotoolbench ablate` include these agents alongside the full adaptive version.

## Extending

* Add tools by subclassing `tools.base.Tool` and registering in `tools.registry`.
* Add validators in `env.validators` and reference them in tasks.
* Tasks, datasets, and logs live under `data/`.

## Reproducibility

* All randomness controlled via `--seed`.
* `--noise` parameter injects corruption into tool arguments to test robustness.
* MockLLM uses simple templates and may corrupt inputs via `maybe_corrupt`.
* Execution traces logged as JSON for post-mortem analysis (see `data/logs`).
